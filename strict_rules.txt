RULE 1: Always use schema-qualified table names in the format "<schema>.table_name".
RULE 2: Do not use "SELECT *"; explicitly list the required columns.
RULE 3: Never filter data using tenant_id; tenant isolation is handled by schema.
RULE 4: When filtering by project title, perform case-insensitive matching using ILIKE.
RULE 5: Use LIMIT 50 by default unless a different limit is explicitly requested.
RULE 6: Avoid using subqueries unless necessary for filtering or aggregation.
RULE 7: When joining tables, always specify join conditions explicitly to prevent Cartesian products.
RULE 8: Do not delete records directly; use an update flag to mark them inactive.
RULE 9: Use COALESCE to handle NULL values in aggregate queries.
RULE 10: When retrieving dates, return them in ISO 8601 format (YYYY-MM-DD).
RULE 11: Always filter out soft-deleted records where applicable.
RULE 12: Use ORDER BY only on indexed columns to optimize performance.
RULE 13: When counting records, use COUNT(*) instead of COUNT(column_name) unless filtering nulls.
RULE 14: For performance, use EXISTS instead of IN when checking related data.
RULE 15: Never expose internal IDs or primary keys unless explicitly requested.
RULE 16: When grouping data, ensure the SELECT clause only contains grouped or aggregated columns.
RULE 17: Always apply security filters before any aggregation.
RULE 18: Avoid returning sensitive fields like passwords, tokens, or API keys.
RULE 19: When retrieving comments or text fields, strip HTML tags before returning.
RULE 20: Ensure SQL queries are safe from injection by using parameterized queries.
